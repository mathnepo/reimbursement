Reimbursement Service - Partner Integration Manual

Overview

The Reimbursement Service API enables third-party reimbursement providers to integrate directly with our Electric Vehicle Supply Equipment (EVSE) management platform. This facilitates efficient and accurate reimbursement for EV charging by providing detailed charging session data in a standardized format.

Typical Integration Flow:

  1. Charge Point Registration: Reimbursement providers register the charge points they want to monitor using this API.
  2. Data Collection: The system collects charging session data from registered charge points, filtered by your specified dates.
  3. Report Generation: Based on your configuration (report frequency and data intervals), the system generates and delivers detailed session reports via webhook.

Reports contain all data occurred since the last report for each registered charge point.

Operation Modes

Reimbursement Service supports two operation modes for managing charge points. You must choose one mode during integration setup:

1. Pooling Mode (Pull Model)

In this mode, Reimbursement Service periodically queries your system to retrieve the complete list of registered charge points.

How it works:

  • Reimbursement Service calls your endpoint at regular intervals
  • Your system returns the list of all currently registered charge points
  • Any charge point not present in the list is automatically considered unregistered
  • No need to explicitly delete charge points, simply omit them from the list

API Request from Reimbursement Service:

GET https://your-api-endpoint.com/charge-points
Authorization: Bearer YOUR_PROVIDED_TOKEN

Note: Any charge point ID not included in this list will be unregistered

Expected Response:

Your endpoint must return a JSON response with the following schema:

{
  "data": [
    {
      "externalId": "chargePoint-1",
      "creationTimestamp": "2021-03-04T12:17:30.000000Z"
    },
    {
      "externalId": "chargePoint-2",
      "creationTimestamp": "2021-09-21T15:06:16.000000Z"
    }
  ]
}

Response Fields:

  • data: Array of charge point objects Return an empty data array if no charge points are registered
  • externalId: Unique identifier of the charge point (serial number)
  • creationTimestamp: ISO 8601 timestamp when the charge point was created and sets the cutoff date for charging sessions.
  • When provided: Only charging sessions occurring after this date will be included in reports
  • When NOT provided: All available historical sessions will be reported

Requirements:

  • Set appropriate creationTimestamp values to control which charging sessions are reported
  • Provide OAuth 2.0 credentials for Reimbursement Service to authenticate

2. Pushing Mode (Push Model)

In this mode, you actively manage charge points by calling our API to add or remove them.

How it works:

  • Use POST /charge-points to register new charge points
  • Use DELETE /charge-points/{serialNumber} to unregister charge points
  • Use GET /charge-points to verify your registered charge points
  • You maintain full control over when changes occur

Requirements:

  • Implement logic to call our API when charge points need to be added/removed
  • Maintain state to track which charge points are registered
  • Handle API responses and potential errors

Authentication

All API requests must be authenticated using OAuth 2.0 Client Credentials via Auth0.

  • Endpoint: https://powerverse-staging.uk.auth0.com/oauth/token
  • Method: POST
  • Headers:
  • Content-Type: application/json

Body:

{
  "client_id": "YOUR_CLIENT_ID",
  "client_secret": "YOUR_CLIENT_SECRET",
  "grant_type": "client_credentials"
}

Example curl:

curl -X POST https://powerverse-staging.uk.auth0.com/oauth/token \
  -H "Content-Type: application/json" \
  -d '{"client_id": "YOUR_CLIENT_ID", "client_secret": "YOUR_CLIENT_SECRET", "grant_type": "client_credentials"}'

Response:

{
  "access_token": "eyJhbGciOiJSUzI1NiIs...",
  "token_type": "Bearer",
  "expires_in": 86400
}

Use the Token

All further requests must include the header:

Authorization: Bearer YOUR_ACCESS_TOKEN

API Base URLs

  • Staging: https://staging-microservices.powerverse.com/reimbursements

API Endpoints

1. Charge Points Management

List Registered Charge Points

GET /v1/charge-points

Optional query parameters: page, perPage

Example:

curl -X GET "https://staging-microservices.powerverse.com/reimbursements/v1/charge-points" \
  -H "Authorization: Bearer YOUR_ACCESS_TOKEN"

Response:

{
  "result": [
    {
      "serialNumber": "MNS000002",
      "creationTimestamp": "2025-04-14T00:00:00"
    }
  ],
  "pagination": { ... }
}

Register a New Charge Point

POST /v1/charge-points

Body:

{
  "serialNumber": "MNS000002",
  "creationTimestamp": "2025-04-14T00:00:00Z"
}

Response: Status 204 No Content

Unregister a Charge Point

DELETE /v1/charge-points/{serialNumber}

Response: Status 204 No Content

2. Configuration Management

Get Current Configuration

GET /v1/config

Response:

{
  "result": {
    "meterDataInterval": 15,
    "reportFrequency": "monthly",
    "callbackUrl": "https://webhook.site-test.com"
  }
}

Update Configuration

PATCH /v1/config

Body:

{
  "meterDataInterval": 30,
  "reportFrequency": "weekly",
  "callbackUrl": "https://your-webhook-url.com/reports"
}

Response: Status 204 No Content

Webhook Reports

When a callbackUrl is configured, the system automatically sends reports based on reportFrequency. Reports include all sessions since the last report, organized per charge point.

Example Report Structure

{
  "chargers": [
    {
      "chargeExternalId": "123456",
      "chargePointExternalId": "chargePoint-1",
      "intervals": [
        {
          "from": "2025-03-07T17:17:02.000Z",
          "to": "2025-03-07T17:30:00.000Z",
          "consumption": 0.772
        }

Understanding Intervals

Energy consumption data is provided in time-based intervals according to your configured meterDataInterval setting:

  • 15 minutes: Most granular option, suitable for detailed analysis
  • 30 minutes: Balanced option, commonly used in the industry
  • 60 minutes: Hourly aggregation for simpler processing

Data Aggregation

Our system automatically aggregates raw charging data into your configured interval size.

For example, if a charging session lasted 45 minutes with 30-minute intervals configured:

  • Interval 1: 0–30 minutes with corresponding consumption
  • Interval 2: 30–45 minutes with corresponding consumption

Consumption Units

All energy consumption values are provided in kilowatt-hours (kWh).

Session Span

Each session starts at the time of plug-in and finishes on unplug. Interval consumption will be 0 for the periods of no consumption.

Notes:

  • Partial sessions at report cut-off are included and completed in following reports.
  • externalId and serialNumber are equivalent identifiers for charge points.

Best Practices

  • Token Management: Tokens expire after 24hrs. Cache and reuse until expiry.
  • Error Handling: Always check for status codes and error details.
  • Validation: Validate data locally before sending.
  • Logging: Log all interactions, but do not log credentials/tokens.
  • Security: Keep client secrets/access tokens confidential.

Frequently Asked Questions

Q: Can I switch between pooling and pushing modes after initial setup? A: No. Switching operation modes requires coordination with our support team to ensure a smooth transition and avoid data loss. Contact support before making such changes.

Q: In pooling mode, how often does Reimbursement Service query our endpoint? A: The pooling frequency is internally managed and can vary depending on infrastructure requirements. It normally oscillates around 30 minutes.

Q: What happens if a charge point is accidentally omitted from the pooling response? A: Any omitted charge point is considered unregistered.

Q: How often are reports generated? A: Reports are generated according to your configured reportFrequency setting:

  • Twice-daily: Reports sent at 00:00 and 12:00 UTC
  • Daily: Reports sent at 00:00 UTC
  • Weekly: Reports sent every Monday at 00:00 UTC
  • Bi-weekly: Reports sent every second Monday at 00:00 UTC
  • Monthly: Reports sent on the 1st day of each month at 00:00 UTC

Q: What happens if my webhook endpoint is unavailable? A: The system will retry failed webhook deliveries with exponential backoff. If all retries fail, you'll need to contact support for manual report retrieval.

Q: Can I change the interval size after registration? A: Yes, you can update your meterDataInterval setting at any time using the /config endpoint. Future reports will use the new interval size.

Q: What timezone is used for timestamps? A: All timestamps are provided in UTC (Coordinated Universal Time) using ISO 8601 format.

Q: Can I register the same charge point multiple times? A: No, each charge point serial number can only be registered once per account. Attempting to register an already-registered charge point will result in an error.

Q: How do I handle partial charging sessions? A: Charging sessions that span multiple reporting periods are not split across reports. Only completed sessions are included in reports. For example, if a session runs from 23:00 to 04:00 and a daily report is generated at 00:00, that session will not be included in the midnight report — it will only appear in the next day's report once the session is completed at 04:00.

Q: Is there a limit to how many charge points I can register? A: There's no hard limit, but we recommend paginating your requests when listing charge points if you have more than 100 registered.
